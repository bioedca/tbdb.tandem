## [Unreleased]

### Added
- **3D similarity cloud (`/cloud`) — an interactive WebGL companion to the 2D similarity map.** The same Stem I sequence-similarity relationships, rendered as a 3D point cloud so the dense Firmicutes core stops being an unreadable flat hairball: the patristic-distance structure is genuinely high-dimensional, and a 3D **classical-MDS / PCoA** embedding recovers **~41 %** of the pairwise distance vs **~32 %** for a flat 2D layout (the shorter antiterminator fallback tree: ~84 % / ~75 %) — both percentages are computed from the embedding's eigen-variance and shown in a standing readout. Same no-polarity guardrail as `/tree` (the `NoPolarityBanner`, and patristic distance is rooting-independent so no ancestry is read), and the same locus ⇄ element / main ⇄ fallback toggles. A new build step (`data-pipeline/build_cloud.py`) computes the embedding offline into a committed `public/data/cloud.json` (847 + 102 points, < 250 KB) with a **deterministic sign convention** so the file is byte-stable across rebuilds; the patristic matrix is computed with a small pure-Python Newick walk (no new `dendropy` dependency). The crowded core is de-piled for clickability by an **anchored-repulsion "spread"** slider that repels nearby points while springing each toward its true PCoA anchor — it is fully **reversible** (spread 0 returns *exactly* to the metric positions), monotone, bounded, and neighbourhood-preserving, and it uses a uniform spatial hash (never an O(n²) scan); the UI surfaces the resulting mean offset and reminds the reader it is "inflated for clarity." Points can be coloured + sized by five vetted **presets** (Specifier · Switch strength · Function · QC/confidence · Taxonomy) or a free color×size **Advanced** disclosure, with a nearest-neighbour **constellation** overlay, a **non-Firmicutes** emphasis (by size, not colour alone), reduced-motion-aware idle rotation, hover tooltips, and click-through to the locus detail page (or specifier cross-filter when used as a dashboard panel). The cloud joins the shared cross-filter (out-of-selection loci dim, never a rebuild). `three` is **dynamically imported** and the route is **code-split**, so neither `three` (732 KB) nor `cloud.json` enters the boot bundle. All point logic (the spread stepper, encodings, element↔locus aggregation, orbit-camera math) lives in pure, unit-tested `src/lib/cloud/*` modules, mirroring how `tree.ts` stays framework-agnostic. Covered by new unit (relax invariants, encodings, aggregation, orbit math, cloud.json data-contract), component (mocked-WebGL smoke), e2e, and data-pipeline (PCoA determinism + committed-artifact integrity) tests. Method note: [`docs/similarity-cloud.md`](docs/similarity-cloud.md).

### Fixed
- **R2DT diagrams are now overlap-free and stay readable at any size / on any device.** The grafted layout kept R2DT's Stem I/II/III coordinates and placed the antiterminator + reflowed single strands by *local* rules with **no global collision check**, so non-adjacent nucleotides overlapped in **85.6%** of diagrams — glyph-on-glyph in the worst cases (e.g. T0281 element 2) — and a handful of elements sprawled into a wide, mostly-empty box that shrank every letter to nothing on desktop. Two fixes: (1) a new **`declash`** stage in `data-pipeline/build_r2dt.py` resolves overlaps with a position-based relaxation that holds the backbone + base-pair rungs rigid and anchors paired (helix) residues hard — so the recognisable straight ladders stay crisp (median helix-rail deviation ~0.1 of a backbone step) while loops / single strands flow apart — driving hard glyph clashes to **0 across all 792 diagrams**; the ~4% of elements that declash still leaves too elongated *or* too sparse (high aspect or low fill) are re-laid on a compact **NAView** layout, keeping the same base pairs so the stem-colour overlay still indexes. (2) A new **`R2dtViewport`** wraps the diagram with a measured, container-aware scaling layer: nucleotide letters now appear automatically whenever the on-screen glyph is large enough (replacing the blunt `n ≤ 360` cutoff), and the reader can **wheel- / keyboard-zoom and drag- / arrow-key-pan** (with a reset) to read any region of an element of any length at a legible scale — independent of figure size, browser, or device. The committed `public/data/r2dt/*.json` are regenerated. Covered by new declash / fill / hard-clash / NAView-gate unit tests, the rewritten antiterminator-ladder test (now asserts an overlap-free, approximately-straight ladder), and the existing component + e2e suites.
- **The R2DT viewer now renders a real antiterminator hairpin and a continuous backbone.** The canonical RF00230 / T-box template models Stem I + the T-box core but **does not base-pair the antiterminator** (it is the alternative regulatory fold, not part of the consensus), so R2DT dumped those ~35 residues into one big unpaired loop ("blob"), left Stem III detached, and stretched the variable inter-stem single strands into backbone "breaks" — which the renderer then *hid* by skipping long segments, making the gaps worse. A new offline post-process — `data-pipeline/build_r2dt.py graft` — keeps R2DT's recognizable Stem I/II/III coordinates but (1) **folds the antiterminator span into a real hairpin** using NAView (ViennaRNA) over the element's own `whole_antiterm_structure` (the same fold fornac shows), oriented to hang off the backbone; (2) draws **one consistent conformation** — it substitutes the antiterminator pairs for R2DT's template pairs that touch that span, so no cross-region rungs leak; and (3) **reflows the single-stranded connectors and tails** (even circular arcs / straight dangles) so the backbone is continuous instead of broken. The frontend draws every backbone segment now (no skip-threshold), since the committed layout is continuous by construction. The raw R2DT output is committed once as `data-pipeline/r2dt_raw.json` so the graft is reproducible without re-running the gated R2DT pipeline; a quality gate drops the **8** elements whose R2DT *base* layout is too distorted to graft cleanly (a pathological exterior loop) to the fornac fallback, so **792 of 949** elements now carry a clean grafted diagram (down from 800 raw, by design). Fixes a latent bug in the loop-arc angle solver along the way (the bisection was inverted, so crammed single strands never bulged). Covered by new data-pipeline tests (pair table, arc reflow, antiterminator fold + template-pair drop, the quality gate, and the driver's drop/clear/manifest behavior).
- **Charts and the similarity tree now sit on white, not the cream page base.** Anything inside a white panel was falling through to the cream page surface: the two dashboard heatmaps (the specifier element-pair matrix and the specifier×phylum grid) set their Plotly `plot_bgcolor` to the cream base, so every empty/zero cell — drawn as a `null`, i.e. "transparent" — and the inter-cell grid gaps rendered cream instead of white (the "0 values default to beige" effect); and the radial similarity-tree canvas used the cream `surface-subtle` token. Every chart `paper_bgcolor`/`plot_bgcolor` and the tree canvas are now explicitly white (`#ffffff`). The cream page background everywhere else (the page base, inputs, sequence/code blocks, table striping, the no-polarity banner) is deliberately unchanged.
- **Scroll-to-zoom in the locus RNA secondary-structure box no longer scrolls the page on wheel-up.** fornac's legacy d3 v3 zoom cancelled the page-scroll default for only one wheel direction, so wheel-down zoomed cleanly but wheel-up zoomed *and* scrolled the whole page. A non-passive `wheel` listener on the fornac mount host now `preventDefault()`s in both directions (fornac still receives the event and zooms; only the page scroll is suppressed), matching the similarity-map tree, which already handles both directions via its d3 v7 zoom.
- **Help (ⓘ) tooltips now actually open.** The `InfoTip` affordances added across the clarity pass carried their definition only on the native `title` attribute, which never fires on keyboard focus or touch and shows nothing on click — so every ⓘ next to a KPI label, chart title, control, or table heading was effectively dead for keyboard and mobile users. `InfoTip` now renders a real, styled popover (the term headword in bold over its definition, on the dark chrome surface so it never reads as a data swatch) shown on hover **and** keyboard focus **and** tap (a tap focuses the button). It is `position: fixed`, so it escapes any `overflow:hidden` card/chart ancestor and renders in full, flips above the trigger when there is no room below, and clamps to the viewport on both axes (so it is never cut off on short/mobile viewports). Escape dismisses it without moving focus (WAI-ARIA tooltip pattern); the popover is `aria-hidden` because the button's `aria-label` already carries the term + definition, so screen readers are not double-read. Covered by a new `InfoTip` component test (hover/focus/Escape, no popover until asked).
- **Corrected the "Discriminator" glossary definition**, which was scientifically wrong — it described the discriminator as "a short conserved element in the antiterminator region" of the riboswitch. The discriminator is a feature of the **tRNA**: the unpaired base just 5′ of its 3′-CCA end, read by the conserved T-box bulge (5′-UGGN-3′) through Watson–Crick pairing. The headword (now **"Discriminator base"**) and definition state this correctly, and stay no-polarity-safe.
- **Final-review accuracy corrections** to the clarity copy above: the `intra-locus pair` definition said identity is scored on **Stem I** — it is actually scored on the **full leader** (Stem I is what the *similarity map* compares); corrected, and reconciled with the `mean %id` definition. The function-class glossary/legend no longer overclaims "amino-acid biosynthesis/transporter" (the classifier buckets by EC class / keyword, which does not establish substrate scope) — now "biosynthetic enzyme (transferase/lyase)" and "membrane transporter/permease". The element-comparison subtitle no longer applies a "more negative" sign reading to **ΔΔG** (kept as a magnitude, per its definition); only the terminator energy carries the directional reading. The heatmap caption uses the canonical verb ("its elements **sense** different amino acids", not "read").

### Added
- **R2DT secondary-structure viewer (canonical RF00230 / T-box template), toggled with fornac.** Each element's detail page now offers a second RNA 2° structure render — an **R2DT** diagram drawn on the canonical **RF00230 / T-box template** (the recognizable, reproducible textbook layout) — switched with a segmented **R2DT ⇄ Fornac** control (R2DT leads where a diagram exists). Both viewers color every nucleotide by its structural domain (Stem I / II / IIA-B / III / antiterminator) from the **same** `color.ts` palette via a shared `buildStemColorMap`, so the two always agree, and both keep the guaranteed tbdb.io VARNA deep-link. The in-app RNA box also moves to a **white** background (matching the chart/tree white-background pass). R2DT is a heavyweight templated pipeline that can't run in the browser, so — like the similarity tree — the diagrams are generated **offline once and committed**: `data-pipeline/build_r2dt.py` emits an R2DT input FASTA and ingests R2DT's RNA-2D-JSON output into compact, recolour-ready per-member assets under `public/data/r2dt/` (per-nucleotide coordinates + base pairs, ~4 KB each) plus an availability manifest; the app fetches one element's diagram on demand and colors it client-side from `stems[]`. The ingest asserts each diagram's sequence equals the member's `fasta_sequence` (T→U), so the color overlay can never misalign; **800 of 949** elements get a faithful full-leader R2DT diagram, and the rest (degenerate or R2DT-clipped leaders) fall back to fornac. Covered by new unit, component, and data-pipeline tests.
- **Member-level base table (`members.csv`) that carries the component-stem spans.** The database now ships a flat, one-row-per-element table (949 rows) alongside the JSON: every per-member field plus the **component-stem identity** flattened into fixed columns — `stem_i_start/end … stem_at_start/end`, 1-based leader-relative, the exact spans the RNA structure is colored by — joined with each element's locus context (accession, strand, organism, phylum). It is emitted by the build (`build_json.py`, committed under `public/data/`) **and** by the downloadable reproduction script (`reproduce_tandem_tbox_db.py`), and offered as a direct download on the About page. This keeps the stem-coloring data **in** the main database rather than in a separate file, and fully reproducible from `Master_tboxes.csv`. Guarded by new artifact tests (949 rows; the `stem_*` columns reconcile with `members.json` `stems[]`; the committed CSV is byte-identical to what the build emits).
- **Per-stem coloring in the locus RNA secondary-structure viewer** — each structural domain of the rendered antiterminator fold is now tinted by which stem it is: **Stem I** (blue), **Stem II** (teal), **Stem IIA/B** (light teal), **Stem III** (purple), and the **antiterminator** (coral), with unpaired linkers a quiet grey and a per-element color key that lists only the domains actually present. The labelled spans are derived in the build (`build_json.py` `derive_stems`) from the Master `s1_*` / `stem2_region_*` / `stem3_*` / `antiterm_*` offsets — the Stem II vs IIA/B split is read from the `other_stems` per-helix list — and emitted as a leader-relative `stems[]` field on every member that indexes the same coordinate frame as `whole_antiterm_structure`. The overlay is applied through fornac's custom-color scheme (which renders non-numeric values as literal fills), inside the existing defensive render path, so any fornac quirk still falls back cleanly to the VARNA deep-link. 791 of 949 elements resolve the full five-domain set; degenerate elements simply omit the missing domains. Mirrored in the public reproduction script (`stems[]` is byte-identical for all 943 shared elements) and covered by new data-pipeline tests.
- **Reproduce-the-database script + an explicit build-method section on the About page** — the About page now documents the full `Master_tboxes.csv` → loci pipeline step by step (drop contamination → orient each T-box on the genome → 600-bp single-linkage clustering → 60-bp collapse to physical cores → the share-a-gene / share-a-specifier / overlapping-leaders tandem test → order 5′→3′ and derive every field), replacing the thinner detection blurb. Alongside it, a new **"Reproduce it yourself"** card offers a self-contained, downloadable Python script — `reproduce_tandem_tbox_db.py`, committed under `public/` and served from the Pages base path — that regenerates the whole dataset from the one public source file. Given `Master_tboxes.csv` it rebuilds `loci/members/identity/summary.json`, the tree-input FASTAs, and an optional readable `tandem_loci.tsv`, then self-verifies on exit (470 / 949 / 488, balanced structures, the CP045927 golden). Reproduction is faithful to the published dataset: the counts, the 394/76 confidence split, and the specifier/phylum distributions match exactly, and **943 of 949 element records are byte-identical**; the few differences (physical cores annotated by several redundant source files, where the published table made a source-specific pick; locus IDs re-assigned in deterministic genomic order) are documented in the script header and the About card. The script's detection stage is a **verified reconstruction** of the upstream locus-detection algorithm, which has no surviving source on disk. Covered by updated About component + e2e tests (new download-link assertions).
- **First-contact primer + shared glossary** — the Dashboard now opens with a one-paragraph, plain-language definition of what a T-box riboswitch and a tandem locus are (a T-box binds one specific tRNA, senses whether it is charged with its amino acid, and switches the downstream gene **on** when that amino acid is scarce; a tandem locus stacks two or more elements in one leader), plus a "New here? → About & method" link. A single-source glossary (`glossary.ts`) defines the load-bearing vocabulary once — present-tense and structural, no evolutionary direction — and is surfaced inline through a small, keyboard-reachable ⓘ affordance (`InfoTip`) so a newcomer can resolve a term in place. The About page gains a "what a T-box is" lead paragraph and plain-language glosses of the detection pipeline (the `core5` 5′ anchor; single-linkage clustering as "chaining nearby cores").

The specificity overview defines "specifier" inline (ⓘ) and adds a legend for the element-pair matrix (each cell = two-element loci by their two specifiers; **diagonal** = same-specifier, **off-diagonal** = mixed; teal = count; `?` = unresolved; folded/symmetric) and the triple-core list (each dot = an element's specifier in 5′→3′ order); the ILE×LEU aside is restated precisely as the largest pairing of two *resolved* specifiers. The specifier×phylum heatmap defines specifier, flags mixed (`A;B`) loci, and drops the loaded word "outliers". - **Editorial pass for terminology + clarity** — all user-facing copy was reviewed against the current T-box literature (Gutiérrez-Preciado 2009; Vitreschak 2008; Marchand 2021 / TBDB) and tightened for concision and precision. Definitions and captions were sharpened throughout — the specifier codon sits "in the Stem I specifier loop"; ΔΔG is "the antiterminator vs terminator free-energy gap"; the operon panel now says the riboswitch senses the **charging state of the cognate tRNA**, not the free amino-acid level; the architecture legend/aria-label read "gene or operon" and "specifier amino acid"; and "EC" is expanded to "Enzyme Commission" on first contact.

## [0.3.0] - 2026-06-03

### Added
- **Similarity-map viewer** (`PhyloTree` on `/tree`) — the real sequence-derived Stem-I tree, rendered with phylotree.js as an **unrooted** equal-angle layout (the stored Newick is midpoint-rooted for stable layout only; no root or polarity is shown or implied). Built element-level (keyed by `unique_name`) and **collapsed to a per-locus default view**: a locus whose element tips form one clade (sisters — a duplication) folds to a single locus tip, while loci whose copies are dispersed stay as separate tips, so the **locus ⇄ element toggle** reveals which. A **main Stem-I ⇄ antiterminator-fallback** tree toggle, a **support-collapse slider** (FastTree SH-like supports; weakly-supported branches fade, default < 0.5), and a **non-Firmicutes-only** filter for the 16 outliers. Tips are coloured by specifier (the primary data axis) with a thin neutral phylum-context ring; hovering a tip shows its locus/element, clicking opens its detail page. phylotree.js (legacy d3 v7 + lodash) is dynamically imported so it stays off the boot path. Verified tip counts — 847 main / 102 fallback elements; 782 / 102 collapsed loci.
- **Tree-load error surface** — the lazy four-artifact tree load now tracks a status and degrades gracefully (the `/tree` view shows a fallback message and a later visit can retry) instead of leaving an unhandled rejection.
- **Persistent no-polarity banner on `/tree`** — the standing `NoPolarityBanner` guardrail now sits above the similarity-map viewer with the exact lab-standard text (*"Exploratory similarity map — unrooted; branch positions reflect sequence similarity, not ancestry."*), pinned as a CI invariant.
- **The tree joins the cross-filtered dashboard** — a similarity-tree panel now sits on the dashboard and shares the one filter state: it **responds** to the shared selection (loci outside the cross-filter dim, so selecting the ILE×LEU cluster narrows the tree to those 10 loci live alongside the table, operon, and heatmap) and **selects** by clicking a tip (narrowing by that locus's specifier, within the existing facet model). The standalone `/tree` route keeps its tip-click-opens-detail behaviour.
- **About / method page** (`/about`) — the provenance and caveats note: how the 470 tandem loci are detected (the upstream contamination-drop → strand/`core5` projection → 600-bp single-linkage → 60-bp collapse → ≥2-distinct-cores pipeline), the data caveats that govern every view (the contamination drop, the corrupt raw-codon column so specifiers come from `amino_acid_top`/`refine_codon_top`, low-confidence loci kept-and-flagged, the near-monochrome taxonomy), the **no-polarity disclaimer** (the similarity map is an exploratory map, not a phylogeny or ancestral-state reconstruction — with the standing banner re-shown and the tree method summarised), the TBDB provenance + citation (CC-BY; Marchand et&nbsp;al. 2021, with the DOI and citing-page links), and a glossary. The load-bearing counts (470 / 949 / 488, confidence, non-Firmicutes) are bound live from `summary.json` rather than hardcoded.
- **End-to-end + visual-regression suite** (Playwright, pinned) — runs against the production build served by `vite preview` at the real Pages base path, across all five routes: the **Dashboard** (KPI values match `summary.json`; selecting the ILE×LEU matrix cell narrows the table, tree, and Sankey to the 10 branched-chain loci; clicking a dashboard tree tip narrows the shared store by specifier), **Browse** (search, facet filter, column sort, CSV export), **LocusDetail** (architecture diagram + element comparison + an RNA tab, with the tbdb.io and NCBI deep-links resolving), **Tree** (renders with the main↔fallback toggle switching trees, the no-polarity banner present), and **About** (the method-note sections and attribution links). Pixel baselines cover the deterministic surfaces only — the dashboard chrome/layout (Plotly + tree masked), the architecture diagram, the similarity tree (fixed seed + layout), and the KPI strip — with the force-directed fornac RNA excluded and a small `maxDiffPixelRatio` absorbing sub-pixel noise. CI's `web` job now installs the pinned browser and runs the suite after the build, uploading the report + snapshots on failure.
